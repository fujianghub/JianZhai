IEEE 802.1X · RFC 5216 · RFC 3748 · RFC 2865 · RFC 8446 · RFC 6960

EAP-TLS 完整认证流程详解

基于 IETF RFC 5216 / 3748 标准，通过双向 X.509 证书认证实现最高安全级别的端口访问控制，涵盖从 EAPoL 初始化到 PTK/GTK 密钥派生的完整协议栈

RFC 3748 · EAP RFC 5216 · EAP-TLS RFC 2865 · RADIUS RFC 8446 · TLS 1.3 RFC 5246 · TLS 1.2 IEEE 802.1X-2020 RFC 3579 · RADIUS/EAP RFC 6960 · OCSP RFC 5280 · X.509 RFC 2759 · MS-CHAPv2 对比
参与组件与职责定义

Client（Supplicant）

运行 IEEE 802.1X Supplicant（Windows 内置 / wpa_supplicant / strongSwan）。持有客户端 X.509 证书 + 私钥（由企业 CA 签发），信任服务器 Root CA。负责发起 EAPoL、完成 TLS 握手双方证书交换、验证服务器证书、导出 MSK。

NAS（Network Access Server）

交换机 / 无线 AP / VPN 网关，充当 IEEE 802.1X Authenticator。对 EAP 内容透明，在 EAPoL 与 RADIUS EAP-Message 属性之间做封装转换。执行端口授权/阻断，安装 VLAN、ACL、QoS 等下发策略，触发 WPA2 四次握手。

RADIUS Server（NPS）

持有服务器 TLS 证书（EKU=ServerAuth，SAN=FQDN），作为 EAP Authentication Server。终结 TLS 握手，验证客户端证书链、EKU、SAN/UPN，对接 AD 完成账户映射和授权策略匹配，派生并下发 MSK。

AD（Active Directory）

存储域账户、安全组、证书模板策略。RADIUS/NPS 通过 LDAP 查询 userAccountControl（账户状态）、证书 UPN 到账户的映射关系、安全组成员关系，用于 NPS 网络策略条件匹配及 VLAN/ACL 授权属性返回。

PKI / CA（AD CS）

企业 CA（通常为 Windows AD CS）为 RADIUS 服务器和所有客户端签发证书。提供 OCSP Responder 和 CRL Distribution Point 服务，供双端实时验证证书吊销状态。Root CA 证书通过 GPO 推送到所有域设备。

阶段一  ·  EAPoL 链路层初始化
01
EAPoL-Start — 客户端触发 802.1X 认证
Supplicant 向 PAE 组播地址发送 EAPoL-Start，NAS 进入认证流程
帧格式（IEEE 802.1X §7.5）
EAPoL Frame · EtherType 0x888E
Dst MAC
01:80:C2:00:00:03
Src MAC
客户端 NIC MAC
EtherType
0x888E
Protocol Version
0x01 (802.1X-2001) 或 0x03
Packet Type
0x01 (EAPOL-Start)
Packet Body Length
0x0000 (无载荷)
端口状态机
  • NAS 端口此时处于 Unauthorized 状态，受控端口（Controlled Port）仅允许 EtherType=0x888E 通过，其他所有流量被丢弃
  • 非受控端口（Uncontrolled Port）始终允许 EAPoL 帧通过
  • 若客户端在链路检测期间未主动发起，NAS 按 txPeriod 定时器周期性发送 EAP-Request/Identity（默认 30 秒）
  • NAS 收到 EAPoL-Start 后将 Identifier 计数器清零，进入 Connecting 状态
⚠ 802.1X-2010 引入 MACsec（802.1AE）可选加密，此时 EAPoL 帧在 MACsec 保护下传输，进一步防止链路层嗅探。
02
EAP-Request/Identity — NAS 请求客户端身份
NAS 向客户端发送 EAP Identity 请求，Identifier 字节用于防重放匹配
EAP 报文结构（RFC 3748 §4）
EAP Packet
Code
0x01 (Request)
Identifier
随机1字节，如 0x7A
Length
0x0005（含 Type 字段）
Type
0x01 (Identity)
  • NAS 不解析 EAP 内容，仅作透明中继（Passthrough Authenticator）
  • 可携带可选的 Type-Data 字段作为提示字符串（如 "Enter username"）
  • Identifier 字段在整个 EAP 会话中单调递增，响应必须携带相同值
03
EAP-Response/Identity — 客户端发送 UPN 身份
Supplicant 回应 Identity，通常为 UPN 格式，NAS 将其封装入 RADIUS Access-Request
Identity 内容格式
  • 用户认证：zhangsan@corp.example.com（UPN 格式，与证书 SAN 中 UPN 一致）
  • 机器认证：host/PC001.corp.example.com（DNS 格式，与证书 SAN 中 dNSName 一致）
  • 匿名保护（可选）：EAP-TLS 本身无需隐藏 Identity（因为 TLS 握手立即跟随），但某些实现支持 anonymous@domain 做 Realm 路由
NAS 封装为 RADIUS
  • NAS 将 EAP PDU 填入 EAP-Message 属性（属性 79），超过 253 字节则拆分为多个属性
  • 同时携带 Message-Authenticator（属性 80）= HMAC-MD5(共享密钥, RADIUS报文)，防止伪造和篡改
RFC 5216 §2.1.1：EAP-TLS Identity Response 中的身份仅用于 RADIUS 路由，实际身份验证由证书中的 CN / SAN 字段决定。
阶段二  ·  RADIUS 接入请求 & EAP-TLS 协商
04

NAS 将 EAP Identity 封装为 RADIUS Access-Request，发往 RADIUS Server（UDP/1812）

05
RADIUS → AD：LDAP 预查询账户存在性
RADIUS/NPS 在 TLS 握手前通过 LDAP 确认账户存在、未被禁用，并获取证书策略配置
LDAP 查询内容
  • 账户查找：以 User-Name 中的 UPN 在 AD 中查找对应 userPrincipalName 属性匹配的账户对象
  • 账户状态：读取 userAccountControl 位标志，确认账户未禁用（ACCOUNTDISABLE=0）、未锁定（LOCKOUT=0）、未过期
  • 安全组：获取 memberOf 属性，用于 NPS 网络策略的组条件匹配
  • 拨入权限：检查 msNPAllowDialin 属性（若使用账户级别控制）
Windows NPS 实现细节
  • NPS 以域计算机账户身份（机器账户 Kerberos 票据）认证到 AD，无需单独服务账户
  • 预查询失败不直接拒绝，TLS 握手后基于证书 UPN 再次进行完整授权验证
06

阶段三  ·  TLS ClientHello
07
TLS ClientHello — 客户端发起 TLS 握手
客户端构造 TLS ClientHello，声明版本、随机数、密码套件、扩展，经 EAP-TLS 封装后传输
TLS 1.3 ClientHello 主要字段（RFC 8446 §4.1.2）
字段内容说明
legacy_version0x0303 (TLS 1.2)TLS 1.3 兼容性占位，实际版本在扩展中协商
random32字节随机数用于派生握手密钥，必须由 CSPRNG 生成
session_id0-32字节TLS 1.3 中为随机值（兼容中间件）
cipher_suites列表如 TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256
supported_versionsTLS 1.3, 1.2扩展 0x002B，指定支持的 TLS 版本
supported_groupsx25519, secp256r1扩展 0x000A，ECDHE 曲线列表
key_shareECDHE 公钥扩展 0x0033，TLS 1.3 中直接发送 DH 公钥
signature_algorithmsRSA-PSS, ECDSA扩展 0x000D，证书签名算法偏好
server_name (SNI)radius.corp.com扩展 0x0000，可选，RADIUS FQDN
EAP-TLS 分片封装
  • TLS ClientHello 通常约 200-400 字节，超过 EAP MTU（默认 1400B）时需分片
  • 分片时首包 L-bit=1（含总长度），中间包 M-bit=1，最后一包 M-bit=0
  • 每收到 M=1 的分片，接收方回复一个空 EAP-Response ACK，发送方继续发送下一片
TLS 1.3 相比 TLS 1.2 的握手往返减少了 1-RTT，在 EAP-TLS 场景下可显著减少认证时延和 RADIUS 报文数量。RFC 9190 定义了 TLS 1.3 在 EAP-TLS 中的使用规范。
阶段四  ·  服务器证书下发 & 客户端验证
08
ServerHello + Certificate + CertificateRequest + ServerHelloDone
RADIUS 回复服务器 TLS 握手消息：选定参数、发送证书链、请求客户端证书
ServerHello（RFC 8446 §4.1.3）
  • server_random：32字节服务器随机数，与 client_random 一起用于密钥派生
  • cipher_suite：从 ClientHello 列表中选定，如 TLS_AES_256_GCM_SHA384
  • key_share（TLS 1.3）：服务器 ECDHE 公钥，双端可计算共享密钥
Certificate（RFC 8446 §4.4.2）
  • 发送 RADIUS 服务器证书 + 所有中间 CA 证书（按顺序，不含 Root CA）
  • RADIUS 服务器证书须满足：EKU=Server Authentication (1.3.6.1.5.5.7.3.1)，SAN 含 dNSName=radius.corp.com，由企业 Root CA 或受信任中间 CA 签发
CertificateRequest（RFC 8446 §4.3.2）
这是 EAP-TLS 与 PEAP 的核心区别：RADIUS 通过 CertificateRequest 要求客户端提供证书，实现真正的双向认证。PEAP 不发送此消息。
  • certificate_types：RSA (1)、ECDSA (64)——可接受的客户端证书类型
  • certificate_authorities：企业 CA 的 Distinguished Name 列表，客户端据此选择匹配的证书
  • signature_algorithms：可接受的签名算法列表
ServerHelloDone（TLS 1.2）/ 等效处理（TLS 1.3）
  • TLS 1.2：发送 ServerHelloDone 标志服务器握手消息完毕
  • TLS 1.3：ServerHello 之后即为 EncryptedExtensions，无 ServerHelloDone
09
客户端验证服务器证书 + OCSP 查询（RFC 6960）
客户端对服务器证书执行完整的五步验证，包括实时 OCSP 吊销状态查询
证书验证五步骤（RFC 5280 §6）
  • ① 信任链：从服务器证书递归验证到受信任的 Root CA（客户端本地信任库，由 GPO 推送）
  • ② 有效期：NotBefore ≤ CurrentTime ≤ NotAfter，时间须与可信 NTP 同步
  • ③ EKU 检查：服务器证书必须含 id-kp-serverAuth (1.3.6.1.5.5.7.3.1)
  • ④ SAN/名称匹配：证书 SAN 中的 dNSName 须与 Supplicant 配置的 RADIUS 服务器名称匹配（严格字符串匹配或通配符匹配）
  • ⑤ 吊销状态：通过 OCSP 或 CRL 验证（见下）
OCSP 请求/响应流程（RFC 6960）
OCSP Request 构成: HashAlgorithm = SHA-1（推荐 SHA-256） issuerNameHash = Hash(证书颁发者 DN) issuerKeyHash = Hash(颁发者公钥) serialNumber = 目标证书序列号 OCSP Response 状态: good → 证书有效，继续握手 revoked → 证书已吊销，终止握手，报告错误 unknown → 无法确定，按策略决定（软失败/硬失败）
OCSP Stapling（RFC 6066 §8）
  • RADIUS 服务器可在 TLS 握手中附带预获取的 OCSP Response（已由 CA 签名），客户端无需自行发起 OCSP 请求
  • 降低握手延迟（减少 1 RTT），保护用户隐私（CA 不知道谁在访问）
  • Windows Server NPS 支持 OCSP Stapling，建议在证书申请模板中启用
⚠ 安全关键：若 Supplicant 未配置受信任的 CA 根证书或未启用服务器名称验证，攻击者可架设伪造 RADIUS 服务器，诱导客户端连接并发送证书，实施 MITM 攻击。
阶段五  ·  客户端证书 + 密钥交换
10
Certificate — 客户端发送证书链
客户端根据 CertificateRequest 中的 CA 列表，选择并发送自身证书及中间 CA 证书
客户端证书要求（RFC 5216 §5.2 + RFC 5280）
  • EKU：必须含 id-kp-clientAuth (1.3.6.1.5.5.7.3.2)
  • 用户证书 SAN：otherName 类型，OID=1.3.6.1.4.1.311.20.2.3，值为 UPN，如 zhangsan@corp.example.com
  • 机器证书 SAN：dNSName 类型，值为计算机 FQDN，如 PC001.corp.example.com
  • 签发者：必须由 RADIUS 侧信任的企业 Root CA / 中间 CA 签发
  • Key Usage：Digital Signature（用于 CertificateVerify）
证书链发送顺序
  • 顺序：客户端证书 → 中间 CA 证书（按颁发链从下到上）→ 不含 Root CA
  • Root CA 由 RADIUS 服务器本地信任库中独立持有，不通过网络传输
若客户端无法找到与 CertificateRequest 中 CA 列表匹配的证书，则发送空 Certificate 消息。RADIUS 须决定是否中止认证（通常配置为拒绝）。
11
ClientKeyExchange — 密钥协商（TLS 1.2）
TLS 1.2 中客户端发送 ClientKeyExchange，完成密钥协商；TLS 1.3 此步骤已在 ClientHello 中完成
TLS 1.2 — ECDHE 模式（RFC 4492）
客户端生成 ECDHE 密钥对: client_private = 随机标量 client_public = client_private × G（椭圆曲线基点） ClientKeyExchange 内容: ECPoint = 客户端 ECDHE 公钥（压缩或非压缩格式） 双端共同计算: 共享密钥 Z = client_private × server_public = server_private × client_public （ECDH 交换结果） pre_master_secret = Z（椭圆曲线点的 x 坐标）
TLS 1.3 — 已在 ClientHello 中完成
  • TLS 1.3 在 ClientHello 的 key_share 扩展中直接发送 ECDHE 公钥
  • 服务器在 ServerHello 的 key_share 中回复服务器 ECDHE 公钥
  • 双端随即计算 (EC)DHE 共享密钥，导出握手密钥，无需单独的 ClientKeyExchange 消息
密钥派生（TLS 1.2 PRF）
master_secret = PRF(pre_master_secret, "master secret", ClientHello.random ∥ ServerHello.random) 输出：48字节
12
CertificateVerify — 客户端证明私钥持有
客户端用证书私钥对所有握手消息的摘要进行签名，向服务器证明自己持有证书对应的私钥
签名构造（RFC 8446 §4.4.3）
TLS 1.3: content = 64个0x20字节 ∥ "TLS 1.3, client CertificateVerify" ∥ 0x00 ∥ Transcript-Hash(所有握手消息) Signature = Sign(client_private_key, content) TLS 1.2: md5_hash = MD5(handshake_messages) sha1_hash = SHA1(handshake_messages) Signature = Sign(private_key, md5_hash ∥ sha1_hash) （RSA-PKCS1v1.5 或 ECDSA）
  • RADIUS 收到后用客户端证书中的公钥验证签名，确认：① 持有私钥 ② 握手消息未被篡改
  • RSA 证书使用 RSA-PSS（TLS 1.3 强制）或 PKCS1v1.5（TLS 1.2 兼容）
  • EC 证书使用 ECDSA 配合对应哈希算法（SHA-256/384/512）
CertificateVerify 是 EAP-TLS 实现客户端向服务器认证的核心消息——证明客户端不仅持有证书（任何人可以复制），还持有与之对应的私钥（无法复制）。
13
ChangeCipherSpec + Finished — 客户端切换加密并验证握手完整性
客户端发送 CCS 切换加密，然后发送 Finished 消息验证完整握手过程的一致性
ChangeCipherSpec（RFC 5246 §7.1）
  • 单字节 0x01，通知对端：后续报文均使用协商的加密参数（密钥+算法）加密
  • TLS 1.3 中 CCS 仅作兼容性占位，实际加密在 ServerHello 后即开始
Finished 消息（RFC 8446 §4.4.4）
TLS 1.3: finished_key = HKDF-Expand-Label(client_handshake_traffic_secret, "finished", "", Hash.length) verify_data = HMAC(finished_key, Transcript-Hash(所有握手消息)) TLS 1.2: verify_data = PRF(master_secret, "client finished", MD5(handshake_msgs) ∥ SHA1(handshake_msgs))[0..11] 输出：12字节
  • RADIUS 验证 verify_data 与本地计算值一致，确认握手过程完整且未被篡改
  • Finished 是 TLS 握手中第一条加密消息（ChangeCipherSpec 之后），验证其 MAC 即等价于验证整个握手历史
阶段六  ·  RADIUS 验证客户端证书（对接 PKI + AD）
14
RADIUS → PKI：OCSP 验证客户端证书吊销状态
RADIUS 向内网 OCSP Responder 查询客户端证书是否已被吊销，这是授权的第一道门
OCSP 查询流程（RFC 6960）
  • OCSP Responder 地址从客户端证书的 Authority Information Access 扩展中读取（id-ad-ocsp OID）
  • RADIUS 构造 OCSP Request，包含：issuerNameHash、issuerKeyHash、serialNumber
  • OCSP Responder 返回带 CA 私钥签名的响应：good / revoked / unknown
吊销场景处理
响应NPS 行为
good继续验证流程
revoked立即返回 Access-Reject，EAP-Failure
unknown取决于 NPS 配置（硬失败：拒绝 / 软失败：继续）
超时/不可达取决于 NPS 策略（生产环境建议硬失败）
备用：CRL 验证
  • 若 OCSP 不可用，RADIUS 从 CRL Distribution Points 扩展下载 CRL 文件
  • CRL 是全量吊销列表，文件较大（MB 级），建议配合 Delta CRL 减少下载量
⚠ 生产环境强烈建议在内网部署专用 OCSP Responder（Windows AD CS 自带），避免依赖外部 CA 服务，防止 OCSP 服务不可达导致认证失败（RADIUS 默认超时 30s）。

完整证书链验证 + EKU + SAN/UPN 检查

RADIUS 侧证书验证详情
  • 信任链：从客户端证书到 RADIUS 信任的 Root CA，每一级签名均须验证
  • 有效期：同客户端验证逻辑

  • SAN/UPN 提取：从 SubjectAltName 扩展的 otherName 字段提取 UPN（OID 1.3.6.1.4.1.311.20.2.3）
  • 吊销状态：步骤14已完成

RADIUS → AD：账户授权策略匹配
NPS 将提取的 UPN 映射到 AD 账户，匹配网络策略，获取 VLAN/ACL 等授权属性
NPS 网络策略匹配逻辑
  • NPS 按策略列表顺序（优先级）逐条检查 条件（Conditions）：用户组、机器组、NAS 类型、时间段、Called/Calling Station ID 等
  • 所有条件满足 → 进入 约束（Constraints） 检查：身份验证方法（必须为 EAP-TLS）、会话超时等
  • 约束满足 → 应用 设置（Settings）：RADIUS 属性（VLAN、Filter-Id 等）

阶段七  ·  服务器 Finished & TLS 握手完成
17
Server ChangeCipherSpec + Finished — 握手双向完成
RADIUS 发送 CCS 和 Finished，客户端验证通过后 TLS 握手正式完成，双向认证成立
Server Finished 验证
TLS 1.3 Server Finished: server_finished_key = HKDF-Expand-Label( server_handshake_traffic_secret, "finished", "", Hash.length) verify_data = HMAC(server_finished_key, Transcript-Hash(所有握手消息)) 客户端验证: 本地重新计算 → 与收到值比较 → 必须完全一致
  • 客户端验证 Server Finished 成功 → 确认握手完整性，TLS 连接建立
  • 此时双端均持有相同的 master_secret（TLS 1.2）或相同的对称密钥（TLS 1.3 AEAD）
  • EAP-TLS 独特要求：客户端发送一个空 EAP-Response（Code=2，Type=13，无 TLS 数据）作为最终 ACK，通知 RADIUS 握手已被客户端确认（RFC 5216 §3.3）
此步骤完成后，双向证书认证正式成立：客户端已验证服务器证书，服务器已验证客户端证书和私钥。整个过程无需用户名/密码参与。
阶段八  ·  MSK 密钥导出 + RADIUS Access-Accept
18
MSK / EMSK 派生 — 从 TLS master_secret 导出会话密钥
RADIUS 从 TLS 握手材料中派生 64 字节 MSK 和 64 字节 EMSK，为后续无线加密提供密钥材料
密钥派生公式（RFC 5216 §2.3）
TLS 1.2 PRF 派生: Key_Material = TLS-PRF(master_secret, "client EAP encryption", client_random ∥ server_random) 输出：128字节 MSK = Key_Material[0..63] （64字节） EMSK = Key_Material[64..127] （64字节） TLS 1.3 HKDF 派生（RFC 9190）: exporter_master_secret 由 TLS Exporter 机制导出 MSK = TLS-Exporter("EXPORTER_EAP_TLS_Key_Material", "", 64字节)[0..63] EMSK = TLS-Exporter("EXPORTER_EAP_TLS_Key_Material", "", 128字节)[64..127]
密钥用途
密钥长度用途
MSK64 字节传递给 NAS，用于派生 WPA2/WPA3 PMK
EMSK64 字节扩展密钥，保留在 RADIUS 侧，用于高级场景（如 EAP Re-auth）
PMK32 字节= MSK[0..31]，802.11 四次握手输入
MSK 完全由 TLS 握手材料派生，与证书内容无关（证书仅用于认证，不直接参与密钥计算）。前向安全性（PFS）由 ECDHE 密钥交换保证——即使 RSA 私钥泄露，历史会话的 MSK 也无法被解密。

阶段九  ·  WPA2 四次握手（无线场景）
20
PMK 就位 → 802.11 四次握手 → PTK/GTK 派生
NAS（AP）和客户端均持有 PMK，通过四次握手派生 PTK（单播加密密钥）和 GTK（组播密钥）
PMK 来源
  • NAS 侧：从 Access-Accept 中解密 MS-MPPE 属性，取 MSK[0..31] 作为 PMK
  • 客户端侧：本地从 TLS master_secret 派生 MSK，同样取 MSK[0..31] 作为 PMK
四次握手密钥派生（IEEE 802.11-2020 §12.7.2）
输入: PMK(32B), ANonce(32B), SNonce(32B), BSSID, STA_MAC PTK = PRF-X(PMK, "Pairwise key expansion", Min(BSSID,STA) ∥ Max(BSSID,STA) ∥ Min(ANonce,SNonce) ∥ Max(ANonce,SNonce)) PTK 拆分（对于 AES-CCMP-128）： KCK = PTK[0..15] (Key Confirmation Key，16字节) KEK = PTK[16..31] (Key Encryption Key，16字节) TK = PTK[32..47] (Temporal Key，16字节) GTK（由 AP 生成，用 KEK 加密后分发给客户端） → 用于保护组播/广播帧
四次握手消息
消息发送方内容
Message 1AP → STAANonce（AP 随机数）
Message 2STA → APSNonce + MIC（用 KCK 计算）
Message 3AP → STAGTK（KEK 加密）+ MIC
Message 4STA → APACK + MIC（确认安装密钥）
四次握手完成后，单播流量使用 TK（AES-CCMP/GCMP）加密，组播使用 GTK 加密。802.11 数据帧的机密性和完整性保护正式生效。
阶段十  ·  端口授权 + 正常通信 + 重认证
21

NAS 端口从 Unauthorized 切换为 Authorized，安装 RADIUS 下发的 VLAN/ACL，客户端开始正常通信
  • NAS 受控端口切换为 Authorized 状态，数据帧按 VLAN ID 转发至对应网段
  • Filter-Id 对应的 ACL 规则生效，执行访问控制（如限制访问特定服务器）
  • NAS 发送 RADIUS Accounting-Start 至 RADIUS，记录会话开始时间和身份信息
Session-Timeout 重认证流程
  • Session-Timeout 到期：NAS 触发 EAP 重认证，重新执行完整 EAP-TLS 流程（或 TLS Session Resumption）
  • 客户端漫游（802.11r/802.11k）：AP 间切换时通过 PMK-R1 缓存跳过完整 EAP，直接进行快速四次握手

TLS 1.3 密钥调度 · RFC 8446 §7
完整密钥派生链：从 ECDHE 到 PTK
▼
TLS 1.3 密钥调度（HKDF 体系）
HKDF-Extract / HKDF-Expand-Label 基本操作: HKDF-Extract(salt, IKM) → PRK HKDF-Expand-Label(Secret, Label, Context, Length): HkdfLabel = Length ∥ "tls13 " + Label ∥ Context → HKDF-Expand(Secret, HkdfLabel, Length) 密钥派生树: 0 (early secret input) │ ├─→ Early Secret = HKDF-Extract(0, PSK or 0) │ └─→ client_early_traffic_secret │ ├─→ Handshake Secret = HKDF-Extract(DHE, Derive(Early)) │ ├─→ client_handshake_traffic_secret → 客户端握手加密密钥 │ └─→ server_handshake_traffic_secret → 服务器握手加密密钥 │ └─→ Master Secret = HKDF-Extract(0, Derive(Handshake)) ├─→ client_application_traffic_secret_0 → 应用数据密钥 ├─→ server_application_traffic_secret_0 → 应用数据密钥 └─→ exporter_master_secret → EAP MSK 派生来源
从 exporter_master_secret 到 EAP MSK
MSK = TLS-Exporter( "EXPORTER_EAP_TLS_Key_Material", // label "", // context（空） 128 // length（字节） )[0..63] EMSK = 同上 [64..127] PMK（802.11）= MSK[0..31]
TLS 1.2 PRF 对比
TLS 1.2 PRF（基于 HMAC-SHA256）: PRF(secret, label, seed) = P_SHA256(secret, label ∥ seed) master_secret = PRF(pre_master_secret, "master secret", ClientRandom ∥ ServerRandom) [0..47] MSK = PRF(master_secret, "client EAP encryption", ClientRandom ∥ ServerRandom) [0..63]
前向安全性保证：每次 TLS 握手生成新的 ECDHE 密钥对（临时密钥），即使 RADIUS 服务器的 RSA/EC 长期私钥泄露，攻击者也无法解密历史会话记录的 MSK，也就无法解密历史无线流量。

RADIUS · RFC 2865 / 3579 / 6614
RADIUS 报文结构与 EAP 封装细节
▼
RADIUS 报文格式（RFC 2865 §3）
RADIUS Packet（UDP）
Code (1B)
1=Access-Req, 2=Accept, 3=Reject, 11=Challenge
Identifier (1B)
请求/响应匹配字节，每次请求递增
Length (2B)
整个 RADIUS 报文长度（含 Header），20~4096 字节
Authenticator (16B)
请求中为随机16B；响应中为 MD5(Code∥ID∥Len∥ReqAuth∥Attrs∥Secret)
Attributes
TLV 格式属性列表（Type 1B + Length 1B + Value）
EAP-Message 属性分片规则（RFC 3579 §3.1）
  • 单个 EAP-Message 属性值最大 253 字节（属性总长 = 253 + 2 = 255 字节）
  • 超过 253 字节的 EAP PDU 需跨多个 EAP-Message 属性顺序存放
  • RADIUS 报文最大 4096 字节，可承载约 16 个 EAP-Message 属性（约 4kB EAP 数据）
  • TLS 证书链传输时通常需要多个 RADIUS 报文往返（每次最多4kB）
Message-Authenticator 计算（RFC 3579 §3.2）
Message-Authenticator = HMAC-MD5( Key = RADIUS 共享密钥, Input = RADIUS 报文（Message-Authenticator 字段置全0） ) 验证规则: - 含 EAP-Message 的报文必须含此属性 - Access-Request 的 Authenticator 为随机数 - 响应报文 Authenticator = MD5(响应报文含 MA ∥ 请求 Authenticator ∥ 共享密钥)
RadSec（RADIUS over TLS，RFC 6614）
  • 在 TLS 1.2/1.3 连接上传输 RADIUS 报文，替代 UDP/1812，解决 RADIUS 明文传输问题
  • 默认端口 TCP/2083，支持双向证书认证（NAS 和 RADIUS 均需证书）
  • 消除 MS-MPPE 属性的 RC4 加密弱点，MSK 在 TLS 保护下传输

EAP-TLS 分片 · RFC 5216 §4.2
分片传输机制与 MTU 协商
▼
为什么需要分片？

TLS 握手消息（尤其是 Certificate 消息，含整个证书链，通常 2~8kB）远超单个 EAP 报文可承载的大小（默认约 1400 字节）。EAP-TLS 通过 L/M-bit 机制实现应用层分片。

分片字段定义
Bit名称发送方行为接收方行为
LLength Included首片置1，包含 TLS 消息总长度（4字节）读取总长度，分配缓冲区
MMore Fragments非末片置1收到M=1时发空ACK，等待下一片
SStartEAP-TLS Start 时置1触发 ClientHello
分片传输流程示例（Certificate，5kB）
→ 片1: EAP-TLS, L=1,M=1, TotalLen=5120, Data[0..1399] ← ACK: EAP-TLS, L=0,M=0, Data=空 → 片2: EAP-TLS, L=0,M=1, Data[1400..2799] ← ACK: EAP-TLS, L=0,M=0, Data=空 → 片3: EAP-TLS, L=0,M=1, Data[2800..4199] ← ACK: EAP-TLS, L=0,M=0, Data=空 → 片4: EAP-TLS, L=0,M=0, Data[4200..5119]（最后一片，M=0） ← 继续正常握手（不回 ACK，直接处理完整 TLS 消息）
MTU 优化建议
  • Supplicant 可通过 EAP-TLS Fragment Size 配置最大分片大小（Windows 默认 1400 字节）
  • 减小证书链大小可减少分片次数：避免在链中包含 Root CA 证书，使用 ECDSA 证书（比 RSA 小约 50%）
  • 服务器端使用 OCSP Stapling，可将 OCSP Response 附加在 Certificate 消息中，减少总 RTT 数

安全分析 · 攻击面 · 加固建议
EAP-TLS 安全威胁模型与防护
▼
EAP-TLS 抵御的威胁
威胁防护机制
密码暴力破解无密码，基于证书私钥（RSA 2048+/ECDSA P-256+），计算上不可破解
重放攻击TLS random + ECDHE 保证每次握手唯一，Finished MAC 覆盖整个握手历史
中间人攻击双向证书认证，攻击者无法伪造任何一方的证书（私钥不可获取）
离线字典攻击不存在密码，无可供攻击的 Challenge/Response
身份泄露TLS 握手后 Identity 通过证书提供，外层 Identity 可设为匿名
流量解密PFS（ECDHE）保证历史流量不可追溯解密
残余风险与缓解措施
  • 证书私钥泄露：攻击者可用泄露的私钥认证。缓解：将私钥存储于 TPM 或智能卡（私钥永不离开硬件），使用 ADCS 自动吊销机制
  • 证书未及时吊销：员工离职后证书仍可用。缓解：实时 OCSP（硬失败策略），AD 账户禁用+证书吊销双重保障
  • RADIUS 共享密钥弱：暴力破解 RADIUS 通信。缓解：使用长随机共享密钥（32+ 字符）或迁移至 RadSec
  • Supplicant 配置错误：未验证服务器证书（虽 EAP-TLS 不像 PEAP 那样危险，但仍可能接受伪造证书）。缓解：通过 GPO 强制配置

EAP-TLS 分片机制

TLS 握手消息超过链路 MTU 时，EAP 层通过 L-bit（携带总长度）和 M-bit（More fragments）进行分片传输。接收端收到 M=1 时回复空 ACK，直至所有分片重组完毕后统一交付 TLS 层处理。

密钥层次结构

pre_master_secret → master_secret → MSK(64B) → PMK(32B) → 四次握手 → PTK/GTK。每层均通过 PRF/HKDF 单向导出，ECDHE 保证前向安全性（PFS）。

证书要求摘要

RADIUS 服务器证书须含 EKU=ServerAuth 及 SAN=FQDN。客户端证书须含 EKU=ClientAuth 及 SAN=UPN（用户）或 DNS（机器）。均需由企业 CA 签发，双端信任同一 Root CA。

OCSP 部署建议

生产环境推荐内网部署 OCSP Responder（AD CS 自带），避免外网 CRL 下载超时（默认 RADIUS 超时 30s）。响应有效期建议 1 小时，刷新间隔 30 分钟，启用 OCSP Stapling 减少握手 RTT。

UPN · EKU · OCSP Stapling

  • UPN（User Principal Name）：证书 SAN otherName 字段，OID 1.3.6.1.4.1.311.20.2.3，RADIUS 用于 AD 账户映射
  • EKU（Extended Key Usage）：X.509 扩展，OID 2.5.29.37，限定证书合法用途，防止跨场景滥用
  • OCSP Stapling：RADIUS 预取已签名 OCSP Response 附在握手中，消除客户端单独查询的延迟和隐私风险

TLS 会话恢复

Session-Timeout 到期重认证时，若双端缓存了 TLS Session Ticket（RFC 5077）或 PSK（TLS 1.3 0-RTT），可跳过完整 TLS 握手，直接恢复会话后导出新 MSK，将重认证 RTT 从 4~6 次降至 1~2 次。
